## Context

The m3u8 stream-processing logic (resolve master‚Üívariant, parse playlist, strip
ads, fetch + rename segments, upload, write playlist, finalize the video row)
lives inside the **io `stream-hls` handler**, wired to `envConfig`, the
configured `hasuraClient`, the logger, and the Hasura-event/Cloud-Task payload
shape. That coupling is *why* the CLI couldn't reuse it and re-implemented the
whole thing inline. Fix the coupling first.

## Scope (behavior-preserving ‚Äî still emits `.ts`)

- Extract a framework-agnostic core, e.g. `src/services/videos/processing/`:
  ```ts
  processStream(input, opts, deps) -> { playlistUrl, duration, ... }
  ```
  with **dependencies injected**, not imported:
  `deps = { storage, http, hasura, logger, ids }`.
  - No direct `envConfig` / `hasuraClient` / `getCurrentLogger` imports inside
    the core ‚Äî callers pass them in.
- Refactor the **io `stream-hls` handler** into a **thin adapter**: validate the
  event ‚Üí build backend `deps` (GCS storage, `fetchWithError`, the configured
  hasura client, logger) ‚Üí call `processStream`.
- **No behavior change**: same `.ts` output, same finalize, same ad-strip,
  custom headers, per-segment resilience. This PR is a pure move + DI.

## Files to touch (ownership)

- `src/services/videos/processing/` (new) ‚Äî the core. **F1 owns it.**
- `src/services/videos/helpers/m3u8/` + `apps/io/videos/routes/stream-hls` ‚Äî
  becomes the thin adapter.
