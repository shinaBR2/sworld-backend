# P1 — Core emits fMP4 (io + CLI get it at once)

**Repo:** sworld-backend
**Type:** feature
**Estimate:** S
**Blocked by:** F1, F2
**Blocks:** V1
**Parallel-safe with:** P2

## Context

With the logic in one place (F1) and both callers on it (io via F1, CLI via F2),
flipping the packaging to fMP4 is a **one-place change** that fixes the actual
bug for both the backend and the CLI.

## Scope

- Change the core's packaging step from `.ts` to **fMP4/CMAF** using the README
  ffmpeg recipe: `-c:v copy`, light audio (`-c:a aac` or
  `-c copy -bsf:a aac_adtstoasc`), `-hls_segment_type fmp4`,
  `-hls_fmp4_init_filename init.mp4`, `%d.m4s`.
- Upload the **init segment** (`init.mp4`, `content-type: video/mp4`) alongside
  `.m4s`; playlist (`#EXT-X-MAP`, `#EXT-X-VERSION:7`) is whatever ffmpeg writes.
- `source` URL + finalize unchanged — only the segment format differs.

## Files to touch (ownership)

- `src/services/videos/processing/` (the core's packaging + upload). **P1 owns
  the fMP4 switch.**

## Acceptance criteria

- [ ] io-processed **and** CLI-processed videos' playlists have `#EXT-X-MAP` +
      `.m4s`.
- [ ] A previously-noisy HE-AAC source (opstream11 family) plays clean in hls.js
      on desktop Chrome, including pause/resume.
- [ ] No video transcode (remux only).

## Test plan

- Unit: core output has `EXT-X-MAP`, init exists, `ffprobe -map 0:a` clean.
- Manual: re-process one real HE-AAC source via both io and CLI → desktop Chrome.

## Out of scope

- convert (P2), catalog migration (G1).
