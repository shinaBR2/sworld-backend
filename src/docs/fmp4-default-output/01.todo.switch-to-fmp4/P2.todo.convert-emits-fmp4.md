# P2 — convert path (mp4→HLS) emits fMP4

**Repo:** sworld-backend
**Type:** feature
**Estimate:** S
**Blocked by:** F1
**Blocks:** V1
**Parallel-safe with:** P1

## Context

The `convert` path (compute service) transcodes an mp4 into HLS. Its
*produce-segments* step is different from the stream path (transcode, not fetch),
but it shares the tail: **package → upload → finalize**. It should reuse the
core's fMP4 packaging/upload (F1) rather than grow its own.

## Scope

- Point convert's HLS output at the core's fMP4 packaging/upload, or apply the
  same fMP4 flags + init-segment upload if convert keeps its own ffmpeg command.
- Video stays `-c:v copy` where it already is; only container/segmenting change.

## Files to touch (ownership)

- `src/services/videos/convert/` — **P2 owns this path.** Reuses F1's
  packaging/upload helper.

## Acceptance criteria

- [ ] A converted video's `source` playlist has `#EXT-X-MAP` + `.m4s`.
- [ ] Plays clean in hls.js on desktop Chrome.
- [ ] Finalize unchanged (only segment format differs).

## Test plan

- Unit/integration: convert a small mp4 → assert fMP4 playlist + clean audio decode.

## Out of scope

- stream/CLI (P1), catalog migration (G1).
