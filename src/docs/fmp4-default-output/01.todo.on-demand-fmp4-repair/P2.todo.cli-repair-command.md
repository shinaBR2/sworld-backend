# P2 — CLI repair command (read GCS `.ts` → repackage → safe swap-in)

**Repo:** sworld-backend
**Type:** feature
**Estimate:** M
**Blocked by:** P1
**Blocks:** V1
**Parallel-safe with:** —

## Context

The operator-facing tool. After a video has streamed in as `.ts`, gone `ready`,
and been judged noisy by a human watching it on the frontend, we run **one
command** to repair just that video into fMP4. The CLI is the natural home: it
already calls the core (F2), it's an operator tool, and "run it when we need" fits
a command you type. No backend endpoint / admin button for now.

The repair reads the **stored `.ts` we own** (option b), never the original
third-party URL — see the epic README for why (availability + same bytes).

## Scope

- New CLI command, e.g. `cli repair-fmp4 <videoId>`:
  1. **Look up** the video (Hasura) → resolve its stored GCS prefix
     (`videos/{userId}/{id}/`) from `source` / id+userId.
  2. **Read** the stored `.ts` playlist + segments from GCS.
  3. **Repackage** via P1's `repackageToFmp4` → `init.mp4` + `.m4s` + playlist.
  4. **Safe swap-in (order matters — swap-in before delete, never a broken
     window; the `source` URL is reused):**
     a. Upload `init.mp4` + `.m4s` **first** (new filenames, no collision with
        the old `.ts`).
     b. **Swap** `playlist.m3u8` (the one shared name) to the fMP4 one, and
        set `no-cache` / invalidate so clients pick it up.
     c. **Verify** the new playlist resolves + a sample decodes clean.
     d. **Only then GC** the old `.ts` objects.
  5. `source` URL is **unchanged** (same `playlist.m3u8` path); the DB row needs
     no update beyond an optional repaired-marker. A failed step never leaves a
     half-deleted video.
- Wire CLI-local deps into P1 (GCS storage, real ffmpeg, console logger), same
  adapter shape as the existing CLI stream command.

## Files to touch (ownership)

- `src/cli/` — **P2 owns the repair command + its dep wiring.**
- May add a small Hasura read (video by id) under `src/services/hasura/queries/`.
- Does NOT modify the existing `stream-m3u8` CLI command or `processStream`.

## Acceptance criteria

- [ ] `cli repair-fmp4 <videoId>` turns that video's stored `.ts` into fMP4 in
      place; `source` URL unchanged; row stays `ready`.
- [ ] Swap-in happens before delete (no broken window); manifest served
      `no-cache`/invalidated.
- [ ] Old `.ts` objects removed only after the new playlist verifies.
- [ ] A failed repackage/upload leaves the original `.ts` video intact.
- [ ] Existing CLI/stream behaviour unchanged.

## Test plan

- Unit/integration: fake GCS + P1 → assert upload-before-swap-before-delete order
  and that a thrown repackage aborts without deleting anything.
- Manual: repair one real noisy video → frontend plays clean on desktop Chrome.

## Out of scope

- Detecting noise automatically (explicitly not doing this).
- Batch repair across many videos (G1).
- Changing default streaming/convert output.
