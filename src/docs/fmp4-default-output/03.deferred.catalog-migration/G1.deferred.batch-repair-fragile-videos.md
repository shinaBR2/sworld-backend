## Status: DEFERRED — do NOT pick up by default.

Existing `.ts` videos **mostly play fine**; the bug is intermittent and only
fragile sources trip it. We fix videos **one at a time, on demand** (P2). Pick
this up **only if** enough videos surface the same noise that one-by-one repair
becomes impractical.

## Context

Once the on-demand repair exists (P1 + P2), "fix many" is just "run the repair
across a list" — not a hand-rolled script. This ticket is that batch wrapper, and
nothing more.

## Scope (if/when prioritized)

- A small batch runner that applies the **P2 repair** to a provided list of video
  ids (prioritise known-fragile sources — HE-AAC / specific CDNs — over a full
  sweep).
- Reuse P2's safe per-video swap-in-before-delete exactly; do **not** reinvent the
  GCS swap logic.
- Run in batches; log what was repaired; verify a sample plays clean.

## Acceptance criteria (per repaired video)

- [ ] `source` playlist has `#EXT-X-MAP` + `.m4s`; old `.ts` removed.
- [ ] Row stays `ready`; `source` URL unchanged.
- [ ] Sample plays clean in hls.js desktop Chrome.

## Out of scope

- The repair engine + command themselves (P1 + P2 — the actual feature).
- Any change to default streaming/convert output.
