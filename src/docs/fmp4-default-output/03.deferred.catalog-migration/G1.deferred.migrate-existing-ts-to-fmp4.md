## Status: DEFERRED — do NOT pick up by default.

Existing `.ts` videos **mostly play fine**; the bug is intermittent and only
fragile sources trip it. Migrating the whole catalog is a large, mostly-cosmetic
batch. Pick this up **only** if more videos surface the same noise.

## Context

Every pre-existing video is `.ts`. The Gosick playlist was already converted
**by hand** during the incident (read each episode's GCS TS playlist → remux to
fMP4 → overwrite, see the incident scripts) — that is the manual template for
this ticket.

## Scope (if/when prioritized)

- A one-off job, per target video. **Order matters — swap-in before delete**, so
  there's never a broken window (the `source` URL is reused):
  1. Read the GCS `.ts` playlist → remux to fMP4 (F1 helper) locally.
  2. **Upload the new `init.mp4` + `.m4s` first** — they have new filenames, so
     they sit alongside the old `.ts` with no collision.
  3. **Swap the playlist** (`playlist.m3u8`, the one shared name) to the fMP4
     one, and **invalidate/​`no-cache` the manifest** so clients pick it up.
  4. **Verify** the new playlist resolves + a sample decodes clean.
  5. **Only then GC the old `.ts`** objects.
  This way a failed remux/upload never leaves a half-deleted video, and no client
  can read a manifest that points at objects that aren't there yet.
- Prioritize known-fragile sources (HE-AAC / specific CDNs) over a full sweep.
- Log what was migrated; do it in batches; verify a sample plays clean.

## Acceptance criteria (per migrated video)

- [ ] `source` playlist has `#EXT-X-MAP` + `.m4s`; old `.ts` removed.
- [ ] Row stays `ready`; `source` URL unchanged.
- [ ] Sample plays clean in hls.js desktop Chrome.

## Out of scope

- Making new processing emit fMP4 (that's F1 + P1–P3 — the actual fix).
