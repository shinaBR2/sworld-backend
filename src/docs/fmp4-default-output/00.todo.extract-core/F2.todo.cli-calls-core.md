## Context

`src/cli/stream-m3u8.ts` re-implements the whole pipeline inline — its own
`slugify` ("Replica of packages/core, kept byte-for-byte"),
`resolveMasterPlaylist`, `parseM3U8Content`, `uploadSegments`, and a raw
`FINALIZE_VIDEO_MUTATION`. Once F1 exists, this duplication should be **deleted**.

## Scope (behavior-preserving — still `.ts`)

- Rewrite the CLI as a **thin adapter** over F1's `processStream`:
  - Parse args (url, `--video-id`, `--referer`, `--standalone`, playlist) →
    build **local** `deps`: GCS storage from the `~/.sworld-cli` key, a hasura
    client from the local endpoint, console logger.
  - Call `processStream` with those deps. Delete the inline
    parse/resolve/upload/finalize/slugify replicas.
- Keep the CLI's UX: interactive prompts, config commands, find-or-create
  playlist linking, standalone mode. Those stay in the CLI (they're CLI
  concerns), but the **processing** is the core's.

## Files to touch (ownership)

- `src/cli/stream-m3u8.ts` — **F2 owns it.** Net deletion of duplicated logic.
- `src/cli/README.md` if behaviour notes change.

## Acceptance criteria

- [ ] CLI imports F1's core; no inline `parseM3U8`/`uploadSegments`/finalize/

- [ ] `npx tsx src/cli/stream-m3u8.ts stream …` produces the same result as
      before (still `.ts` at this point).
- [ ] CLI still runs standalone (no backend boot, local config).
