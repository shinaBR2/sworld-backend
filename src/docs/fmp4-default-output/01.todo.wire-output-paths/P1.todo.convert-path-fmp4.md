# P1 — convert path (mp4→HLS) emits fMP4

**Repo:** sworld-backend
**Type:** feature
**Estimate:** S
**Blocked by:** F1
**Blocks:** V1
**Parallel-safe with:** P2, P3

## Context

The `convert` path (compute service) already shells out to ffmpeg to turn an
mp4 into HLS, so this is the cheapest path to flip — it's a flag change plus
calling F1's helper for upload.

## Scope

- Switch the convert ffmpeg HLS output to fMP4 via F1's `packageToFmp4` (or the
  same flags inline if convert builds its own ffmpeg command):
  `-hls_segment_type fmp4 -hls_fmp4_init_filename init.mp4 -hls_segment_filename %d.m4s`.
- Upload `init.mp4` + `.m4s` + playlist via F1's upload support.
- Video stays `-c:v copy` where it already is; only the container/segmenting
  changes.

## Files to touch (ownership)

- `src/services/videos/convert/` (the ffmpeg/HLS step) — **P1 owns this path.**

## Acceptance criteria

- [ ] A converted video's `source` playlist has `#EXT-X-MAP` + `.m4s`.
- [ ] Plays clean in hls.js on desktop Chrome (the failing environment).
- [ ] `tasks`/`videos` finalize unchanged (only segment format differs).

## Test plan

- Unit/integration: convert a small mp4 locally → assert fMP4 playlist + clean
  `ffprobe` audio decode.

## Out of scope

- stream-hls (P2), CLI (P3), catalog migration (G1).
