# P2 — stream-hls / m3u8 re-stream remuxes to fMP4

**Repo:** sworld-backend
**Type:** feature
**Estimate:** M
**Blocked by:** F1
**Blocks:** V1
**Parallel-safe with:** P1, P3

## Context

This is the path that **had the bug**. The m3u8 re-stream today is pure
**passthrough**: it fetches the source `.ts` segments, strips ads, renames them
`0.ts, 1.ts, …` and uploads them as-is — no ffmpeg. So whatever AAC/ADTS quirk
the source has, we ship it, and hls.js trips on it.

## Scope

- Insert a **TS→fMP4 remux** step (via F1's `packageToFmp4`) before upload:
  feed the (ad-stripped) segment set through ffmpeg, emit `init.mp4` + `.m4s`.
  - Low-CPU: `-c:v copy`, light audio (`-c:a aac` or `-c copy -bsf:a aac_adtstoasc`).
- Keep the existing ad-strip / segment-selection logic; only the final
  packaging + upload changes from "copy `.ts`" to "remux → fMP4".
- Preserve per-segment fetch resilience / custom request headers already in this
  path.

## Files to touch (ownership)

- `src/services/videos/helpers/m3u8/` (the streamM3U8 logic) and the
  `apps/io/videos/routes/stream-hls` handler — **P2 owns this path.**

## Acceptance criteria

- [ ] A re-streamed video's `source` playlist has `#EXT-X-MAP` + `.m4s`.
- [ ] A previously-noisy source (HE-AAC, e.g. the opstream11 family) plays clean
      in hls.js on desktop Chrome, including pause/resume.
- [ ] Ad-stripping + custom headers still work.
- [ ] No video transcode (remux only).

## Test plan

- Unit: stream a small TS fixture → assert fMP4 output, clean audio decode.
- Manual: re-process one real HE-AAC source, play on desktop Chrome.

## Out of scope

- convert (P1), CLI (P3), catalog migration (G1).
