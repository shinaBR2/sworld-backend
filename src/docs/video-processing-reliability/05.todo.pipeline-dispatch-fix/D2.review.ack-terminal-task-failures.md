# D2 — ACK terminal task failures (stop the infinite retry)

**Repo:** sworld-backend
**Type:** bug (production)
**Estimate:** XS
**Blocked by:** D1
**Blocks:** —

## The incident

After D1 fixed dispatch, the pipeline ran for real — and a hotlink (403) source
failed correctly (`status='failed'` + Slack alert). But the **Cloud Task never
left the queue**: it retried forever (12, 9, … attempts) for a *permanent* 403.

## Root cause

`honoRequestHandler` has no error handling — it just `c.json(await handler())`.
So when a handler throws (the terminal 403, re-thrown by `withVideoFailureReport`)
the error propagates to Hono's default handler → **HTTP 500** → Cloud Tasks treats
it as retryable and retries indefinitely.

A terminal, non-retryable failure is permanent: retrying can't help, and each
retry re-runs `markVideoFailed`, which re-fires the `video_on_failed` event →
**duplicate Slack alerts** for the same video.

## The fix

In `withVideoFailureReport`, after recording the failure:

- **Terminal** (`CustomError` with `shouldRetry === false`): **ACK with a 2xx**
  (`AppResponse(false, message)`) so Cloud Tasks drops the task. The video is
  already `failed` and the alert already fired — exactly once.
- **Retryable**: **re-throw** (→ 5xx) so Cloud Tasks retries transient blips.

The Slack alert is unaffected: it's driven by the `markVideoFailed` →
`status: 'failed'` transition, which happens *before* we ACK.

## Files to touch

- `src/middleware/reportVideoFailure/index.ts` — ack terminal, re-throw retryable.
- Its test.

## Acceptance criteria

- [ ] Terminal failure → video `failed`, one Slack alert, task **leaves** the queue (2xx).
- [ ] Retryable failure → re-thrown (5xx), Cloud Tasks still retries.
- [ ] No change to non-video handlers (this lives in the video-task wrapper).

- Geo/IP-blocked sources (e.g. opstream/quaivat 403 from GCP egress) — an infra
