# D3 — Deploy on shared-code changes (CI/CD path filters)

**Repo:** sworld-backend
**Type:** bug (CI/CD)
**Estimate:** XS
**Blocked by:** —
**Blocks:** shipping D2 (and any future shared-code fix)

## The incident

Merging D2 (#161) to `main` triggered **no deploy** — so the queue-ack fix sat in
`main` without ever running in PROD (the Cloud Tasks queue kept retrying).

## Root cause

Each of the 6 deploy workflows only fires on changes under its own
`src/apps/<service>/**` (+ that service's Dockerfile + `package.json`):

```yaml
paths:
  - 'src/apps/io/**'   # ← misses shared code
```

But every service image bundles the **shared** code too — `src/middleware/`,
`src/utils/`, `src/services/`, `src/schema/`, `src/types/`. D2 changed
`src/middleware/reportVideoFailure/**`, which matched no filter → no deploy. Any
shared-code change is silently never deployed. There was also no
`workflow_dispatch`, so no way to redeploy manually.

(D1 deployed only because it touched `src/apps/gateway/**`; its shared validator
change didn't redeploy io/compute.)

## The fix

For all 6 deploy workflows (dev + prod × io/compute/gateway):

1. Broaden the push path filter to **`src/**`** (+ keep Dockerfile + package
   files). Over-triggers slightly — a gateway-only change also redeploys io and
   compute — which is safe and far better than silent non-deploys. Surgical
   per-dir lists are fragile.
2. Add **`workflow_dispatch:`** so a service can be redeployed manually (needed to
   ship an already-merged shared-code change like D2, and useful generally).

## Acceptance criteria

- [ ] A change under `src/middleware|utils|services|schema/**` triggers the
      relevant service deploys.
- [ ] Each deploy workflow can be run manually from the Actions tab.

## Follow-up (manual, after merge)

- Trigger **prod-io** + **prod-compute** via `workflow_dispatch` to finally ship
  D2's queue-ack fix (already-merged history won't auto-trigger).

## Out of scope

- Optimising which services rebuild per change (a shared change rebuilds all
  three). Correctness first; can be made surgical later if deploy volume matters.
