# D1 — Fix task payload shape (pipeline dispatch) + fail-loud validation

**Repo:** sworld-backend
**Type:** bug (production)
**Estimate:** S
**Blocked by:** —
**Blocks:** Z1 (and the whole reliability epic working in PROD)

## The incident

The auto-processing pipeline has been **silently dead in PROD since ~2026-04-03**.
Every video added through the app stays at `status='processing'` forever; no
alert, no error. Videos have only been getting processed via the manual CLI.

Discovered while trying to verify Z1: a new video produced a `stream_hls` task
(gateway works) and the io service even returned **HTTP 200** — but in ~16 ms with
a 489-byte body. The handler never actually ran.

## Root cause

**Payload shape mismatch.** The gateway (`stream-to-storage`) sends the **raw
snake_case** Hasura event as the task body:

```ts
const { data, metadata } = transformEvent(event); // camelCase computed…
// …but then:
payload: event,                                    // …raw snake_case sent
```

The io / convert / import handlers validate against a **camelCase** schema, so
validation fails on every task. Reproduced against the real payload:

```
body data userId is required        (payload has user_id)
body data videoUrl is required      (payload has video_url)
body data keepOriginalSource is required   (payload has keep_original_source)
body metadata spanId is required    (payload has span_id)
body metadata traceId is required   (payload has trace_id)
```

**Why it was silent for ~2 months:** `honoValidateRequest` returns **HTTP 200**
(with a `{success:false}` envelope) on a validation failure. Cloud Tasks sees a
2xx → "delivered", no retry, no dead-letter; the real handler never runs and
**nothing throws**, so `withVideoFailureReport` never marks the video `failed` and
no Slack alert fires. A shape mismatch became an invisible outage.

## Scope

1. **Fix dispatch:** the gateway sends the **transformed** payload it already
   computes — `payload: { data, metadata }` — so handlers receive the camelCase
   shape their schemas require. Repairs stream-hls, convert, and import at once.
2. **Fail loud (hardening):** `honoValidateRequest` returns **HTTP 400** (not 200)
   on validation/parse failure, so Cloud Tasks retries/dead-letters and the
   failure is visible instead of silently swallowed. Prevents this whole class of
   bug from hiding again.

## Files to touch

- `src/utils/validators/request.ts` — 200 → 400 on failure.

## Acceptance criteria

- [ ] Gateway sends `{ data, metadata }` (camelCase) as the task payload.
- [ ] The real task payload validates against the io stream-hls schema (verified
      with `transformEvent(event)` → `streamHandlerSchema` = success).
- [ ] Validation/parse failures return `400`, not `200`.
- [ ] Existing gateway + validator tests updated and green.

## Test plan

- Unit: gateway payload assertions + validator 400 assertions.
- PROD verify (also unblocks Z1): add a video → it processes to `ready`
  (or `failed` + Slack on a hotlink source) instead of hanging at `processing`.

## Out of scope

- Backfilling the videos already stuck at `processing` (separate follow-up).
- Marking a video `failed` on a *validation* error (validation happens before the
  handler; a malformed task now dead-letters via the 400 instead).
