# M1 — Migrate `cloud-task` to Hasura task ops (drop transaction)

**Repo:** sworld-backend
**Type:** migration (call site)
**Estimate:** S
**Blocked by:** H1
**Blocks:** D1
**Parallel-safe with:** M2, M3, M4, M5, M6

## Context

`createCloudTasks` is the only consumer of `createTask` + `updateTaskStatus`. It
currently wraps `createTask` → (external GCP `client.createTask`) →
`updateTaskStatus` in a Sequelize transaction. The external GCP call in the
middle can't participate in a DB transaction, so the transaction was never fully
sound. Swap to the Hasura task ops from H1 and remove the transaction — safe
because `createTask` is idempotent (upsert keyed by the `uuidv5` task id), so a
retried run finds the existing task and proceeds. See the epic README "lost
transaction" note.

## Scope

- Replace `createTask` / `updateTaskStatus` imports from `src/database/queries/tasks`
  with the H1 Hasura helpers (`src/services/hasura/mutations/tasks`).
- Remove `const transaction = await sequelize.transaction()`, the
  `transaction.commit()` / `transaction.rollback()` calls, and the `transaction`
  argument threaded into the task helpers. Drop the `sequelize` import.
- Preserve existing control flow exactly: create (upsert) task → if
  `dbTask.completed` return `null` → create GCP cloud task → set task
  `IN_PROGRESS`. Keep the same error log fields and re-throw.
- Update `cloud-task.test.ts` to mock the Hasura helpers instead of Sequelize /
  the transaction; assert the early-return-on-completed and error paths still hold.
